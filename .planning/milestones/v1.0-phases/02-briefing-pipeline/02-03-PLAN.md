1. Credential regex pattern (per D-09). Addresses review concern: use a bounded capture `[\S]{1,200}` instead of unbounded `\S+` to prevent catastrophic backtracking, and add common delimiters as stop characters to avoid mangling JSON/HTML:
   ```python
   CREDENTIAL_PATTERN = re.compile(
       r'(?:'
       # Key-value patterns: password: xxx, token=xxx, etc.
       # Capture the value up to a delimiter (comma, quote, angle bracket, whitespace, or end of string)
       r'(?:password|passwd|token|api_key|apikey|secret|auth|authorization|bearer)'
       r'\s*[:=]\s*'
       r'[^\s,;"\'>]{1,200}'
       r'|'
       # URL auth params: ?token=xxx&, ?key=xxx
       r'https?://[^\s<>"]{0,500}[?&](?:token|key|secret|auth|session_id)=[^\s<>"&]{1,200}'
       r')',
       re.IGNORECASE
   )
   ```
   Key changes from original:
   - Value capture uses `[^\s,;"\'>]{1,200}` instead of `\S+` — stops at comma, semicolon, quotes, angle brackets (JSON/HTML delimiters)

   - URL pattern uses `[^\s<>"&]{1,200}` for the param value — stops at `&` for next param

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Raw body -> redactor | Untrusted email/message content enters redactor for sanitisation |
| Redactor -> narrator LLM | Summarised + credential-stripped text crosses to GPT-4.1 context |
| Narrator LLM -> backend | Structured JSON output consumed by backend only |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-06 | Tampering (Prompt Injection) | redactor.py | mitigate | Raw body is summarised by GPT-4.1 mini into "key actionable facts" — this reduces the attack surface. The summary is then regex-stripped of credentials. The narrator LLM receives only the summary, not the raw body. System/user message separation in both LLM calls. |
| T-02-07 | Information Disclosure | redactor.py | mitigate | CREDENTIAL_PATTERN regex with bounded capture strips password, token, api_key, secret, Authorization, and URL auth params before any content reaches the narrator LLM. Per D-09/D-10, redaction is per-item. Bounded capture `{1,200}` prevents regex backtracking. |
| T-02-08 | Elevation of Privilege | narrator.py | mitigate | Narrator LLM has NO tools registered, NO function_call parameter, NO credentials in context. response_format=json_object constrains output. Backend parses JSON, validates "narrative" key exists, and extracts only that key. Per D-11/SEC-05. |
| T-02-09 | Information Disclosure | narrator.py | accept | LLM provider (OpenAI) sees summarised content. Accepted for M1 — OpenAI data usage policy applies. Full PII detection deferred to M2 (presidio). |
| T-02-15 | Denial of Service | redactor.py | mitigate | Concurrent OpenAI calls capped by asyncio.Semaphore(3) to prevent rate limit exhaustion. Individual call failures handled gracefully. |
</threat_model>
