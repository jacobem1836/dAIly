# Plan 07-03: Bug C — Resolve message_id from briefing context (no more last_content stub)

## Objective

Implement D-05 (replace raw content passthrough with LLM-assisted extraction) and D-06
(briefing item metadata is already wired to the orchestrator state via `email_context` —
this plan consumes it correctly).

Currently `summarise_thread_node` at `nodes.py:227` does `message_id = last_content` — it
passes the user's literal utterance to `adapter.get_email_body(message_id)`, which only
works if the user types an exact message ID. This is brittle and silently broken for any
natural-language query like "summarise the email from Paul about the internship".

The fix: state already carries `email_context: list[dict]` populated at session init by
`session.py` (already exists, see state.py:50). Add an LLM-assisted matcher that picks the
correct message_id from that list given the user's free-text reference.

Purpose: Correctness fix — the thread-summarise feature works for any natural reference,
not just literal IDs.

Output: New helper `_resolve_thread_target(user_query, email_context, client)` calls
GPT-4.1 mini to map user intent to a real `message_id`; `summarise_thread_node` uses it
before calling `get_email_body`.

2. Add a new module-level async helper directly above `summarise_thread_node`:

   ```python
   async def _resolve_thread_target(
       user_query: str,
       email_context: list[dict],
       client: AsyncOpenAI,
   ) -> str | None:
       """Match a user's natural-language reference to a real message_id.

       Calls GPT-4.1 mini with the formatted email_context list and the user's
       request. Returns the matched message_id, or None if no email matches or
       the LLM call fails.

       Args:
           user_query: The user's last message content (free text).
           email_context: List of email metadata dicts from SessionState.email_context.
           client: AsyncOpenAI client.

       Returns:
           The matched message_id, or None if unmatched / error.
       """
       if not email_context:
           return None
       try:
           prompt = THREAD_TARGET_RESOLVER_PROMPT.format(
               email_context=_format_email_context(email_context),
               user_query=user_query,
           )
           response = await client.chat.completions.create(
               model="gpt-4.1-mini",
               messages=[
                   {"role": "system", "content": prompt},
                   {"role": "user", "content": user_query},
               ],
               response_format={"type": "json_object"},
               max_tokens=80,
           )
           parsed = json.loads(response.choices[0].message.content)
           candidate = parsed.get("message_id")
           if not candidate or candidate == "NONE":
               return None
           # Validate the candidate exists in email_context — never trust LLM output blindly
           valid_ids = {e["message_id"] for e in email_context}
           return candidate if candidate in valid_ids else None
       except Exception as exc:
           logger.warning("_resolve_thread_target: failed: %s", exc)
           return None
   ```
