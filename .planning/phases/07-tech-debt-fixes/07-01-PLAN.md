# Plan 07-01: Bug A — Resolve user_email from connected integration

## Objective

Replace the `user_email = ""` stub in `_build_pipeline_kwargs` (scheduler.py:84) with a real
lookup against the connected Google or Microsoft account so `_is_direct_recipient()` in
`ranker.py` can match the user against the email `To:` header and award `WEIGHT_DIRECT` (10).

Implements D-01 (extract from connected integration) and D-02 (graceful fallback when no
email integration is connected).

Purpose: WEIGHT_DIRECT currently never fires — every briefing silently under-prioritises
emails directly addressed to the user. This is a correctness fix to the ranker input, not a
new feature.

Output: `scheduler.py` resolves `user_email` from the first available Google or Microsoft
token; tests cover both providers and the empty-fallback path.

<action>
Edit `src/daily/briefing/scheduler.py`:

1. Add a module-level async helper directly above `_build_pipeline_kwargs` with this exact signature and behaviour:

   ```python
   async def _resolve_user_email(
       provider: str,
       google_creds: "Credentials | None" = None,
       outlook_token: str | None = None,
   ) -> str:
       """Resolve the authenticated user's primary email from a provider token.

       Returns empty string on any failure — caller treats empty as "skip WEIGHT_DIRECT" (D-02).

       For provider == "google": calls Gmail API `users().getProfile(userId="me")`
       inside `asyncio.to_thread` and returns the `emailAddress` field.

       For provider == "microsoft": calls Microsoft Graph `/me` endpoint via httpx
       (Authorization: Bearer <outlook_token>) and returns the `mail` field, falling back
       to `userPrincipalName` if `mail` is None.
       """
   ```

   Implementation details:
   - Google branch: `from googleapiclient.discovery import build`; build `gmail` v1 service from `google_creds`; call `service.users().getProfile(userId="me").execute()`; return `result.get("emailAddress", "")`. Wrap the whole sync block in `asyncio.to_thread`.
   - Microsoft branch: use `httpx.AsyncClient` with `headers={"Authorization": f"Bearer {outlook_token}"}` to GET `https://graph.microsoft.com/v1.0/me`; on 200 return `data.get("mail") or data.get("userPrincipalName") or ""`; on any non-200 return `""`.
   - Wrap the entire body in `try/except Exception` returning `""` so a profile-API failure never crashes the pipeline.
   - Lowercase the result before returning (`.lower().strip()`) so it matches `_is_direct_recipient()` casing.

2. In `_build_pipeline_kwargs`, replace the stub initialisation:
   - Line 84 currently: `user_email = ""`
   - Replace with: `user_email = ""` (keep the default) AND inside the existing `for token in tokens:` loop, after each provider branch is fully constructed, call the resolver if `user_email` is still empty:
     - In the `google` branch: after `calendar_adapters.append(cal)`, add:
       ```python
       if not user_email:
           user_email = await _resolve_user_email("google", google_creds=google_creds)
       ```
     - In the `microsoft` branch: after `email_adapters.append(outlook)`, add:
       ```python
       if not user_email:
           user_email = await _resolve_user_email("microsoft", outlook_token=decrypted)
       ```
   - Do NOT add a Slack branch — Slack does not give us a primary email and is not in scope.

3. The existing `return` block at line 127 already passes `"user_email": user_email` — leave that untouched.

4. Add `import httpx` at the top of the file with the other imports.
</action>
