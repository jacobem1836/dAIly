**Dockerfile** (per D-04, D-05, D-06):

Create a multi-stage Dockerfile at repo root:

1. **Stage 1 (uv-source):** `FROM ghcr.io/astral-sh/uv:latest AS uv-source` -- source for the uv binary only.

2. **Stage 2 (builder):** `FROM python:3.11-slim AS builder`
   - `COPY --from=uv-source /uv /uvx /bin/`
   - `WORKDIR /app`
   - Copy `pyproject.toml` and `uv.lock` first (layer caching -- deps cached when only src changes)
   - `RUN uv sync --frozen --no-dev --no-install-project` (install deps without the project itself)
   - Copy `src/` directory
   - `RUN uv sync --frozen --no-dev` (install the project package into the venv)

3. **Stage 3 (final):** `FROM python:3.11-slim AS final`
   - `WORKDIR /app`
   - Copy `.venv` from builder
   - Copy `src/` from builder
   - Copy `alembic/` directory and `alembic.ini` (needed for entrypoint migrations)
   - Copy `scripts/entrypoint.sh`
   - Set `ENV PATH="/app/.venv/bin:$PATH"`
   - `RUN chmod +x /app/entrypoint.sh`
   - `EXPOSE 8000`
   - `ENTRYPOINT ["/app/entrypoint.sh"]`

No `CMD` needed -- the entrypoint script handles everything.

**scripts/entrypoint.sh** (per D-05):

Create `scripts/entrypoint.sh`:
```bash
#!/bin/sh
set -e
echo "Running database migrations..."
alembic upgrade head
echo "Starting uvicorn..."
exec uvicorn daily.main:app --host 0.0.0.0 --port 8000
```

- `set -e`: If migrations fail, the container does not start.
- `exec`: Replaces the shell so uvicorn receives SIGTERM for graceful shutdown.
- App listens on `0.0.0.0:8000` per D-06.

**alembic/env.py patch** (critical pitfall from RESEARCH.md):

Patch both `run_migrations_offline()` and `run_async_migrations()` to read `DATABASE_URL` from the environment before falling back to `alembic.ini`:

```python
import os
# In both functions, replace:
#   url = config.get_main_option("sqlalchemy.url")
# with:
url = os.environ.get("DATABASE_URL", config.get_main_option("sqlalchemy.url"))
```

Add `import os` at the top of the file (after the existing imports). This is a one-line change per function that makes migrations work inside Docker where the Postgres host is `postgres` (the compose service name), not `localhost` as hardcoded in `alembic.ini`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Host -> Container | .env file with secrets passed into container runtime |
| Container -> Postgres | Database credentials in DATABASE_URL |
| Public Internet -> Port 8000 | App port exposed on host (loopback-only on VPS via DEPLOY.md) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-15-01 | Information Disclosure | Dockerfile | mitigate | Never COPY .env into image; .dockerignore excludes .env; secrets passed via env_file at runtime only |
| T-15-02 | Information Disclosure | .git in image | mitigate | .dockerignore excludes .git directory; no repo history in container |
| T-15-03 | Tampering | entrypoint.sh | mitigate | entrypoint uses set -e to fail fast on migration errors; exec replaces shell for proper signal handling |
| T-15-04 | Denial of Service | App startup race | mitigate | depends_on with condition: service_healthy ensures postgres and redis accept connections before app starts |
| T-15-05 | Elevation of Privilege | Container runs as root | accept | M1 single-host deployment; non-root user is a v2.0 hardening item. No PII exposure from root in container. |
</threat_model>
